# JADX MCP 406 错误和连接慢问题修复

## 问题现象

从日志中看到：
1. **大量 406 Not Acceptable 错误**：服务器启动后，检查函数返回大量 406 错误
2. **连接慢**：需要等待很久才能成功连接（约 20-30 秒）

## 根本原因

### 1. `checkJADXServerRunning` 缺少 `Accept` 请求头

**问题代码（修复前）：**
```go
req.Header.Set("Content-Type", "application/json")
// 缺少 Accept 头！
```

**FastMCP 服务器要求：**
- FastMCP streamable-http 传输模式需要客户端发送 `Accept: application/json, text/event-stream` 头
- 如果没有这个头，服务器返回 `406 Not Acceptable`

**日志证据：**
```
[32mINFO[0m:     127.0.0.1:24749 - "[1mPOST /mcp HTTP/1.1[0m" [31m406 Not Acceptable[0m
```

这些 406 错误都是来自 `checkJADXServerRunning` 的频繁检查（每 200-500ms 一次）。

### 2. 频繁的服务器检查导致大量无效请求

**问题：**
- `checkJADXServerRunning` 在服务器启动期间每 200-500ms 调用一次
- 每次调用都发送 HTTP 请求，但由于缺少 Accept 头，返回 406
- 在 10 秒的等待期内，可能发送 20-50 个无效请求

**影响：**
- 产生大量日志噪音
- 浪费网络资源
- 可能影响服务器性能

## 修复方案

### 1. 添加 `Accept` 请求头（关键修复）

```go
// checkJADXServerRunning - 修复后
req.Header.Set("Content-Type", "application/json")
req.Header.Set("Accept", "application/json, text/event-stream")  // ✅ 添加这行
```

### 2. 优化检查逻辑

**将 406 视为服务器运行状态：**
```go
// 406 Not Acceptable 表示服务器在运行但请求格式不对
// 添加 Accept 头后，应该不会再返回 406，但保留判断以防万一
return resp.StatusCode == 200 || resp.StatusCode == 400 || 
       resp.StatusCode == 500 || resp.StatusCode == 406
```

### 3. 使用渐进式检查间隔

**优化前：**
```go
checkInterval := 500 * time.Millisecond  // 固定间隔
```

**优化后：**
```go
// 渐进式检查：开始时频繁，后面逐渐减少
// 前3次：200ms
// 4-10次：500ms
// 11次以后：1秒
```

这样可以：
- 快速检测服务器是否启动（前几秒）
- 减少无效请求数量（后期）
- 降低服务器负载

## 修复效果

### 修复前：
- ✅ 服务器启动成功
- ❌ 大量 406 Not Acceptable 错误（20-50个）
- ❌ 连接耗时：~20-30 秒
- ❌ 检查请求：20-50 次（大部分返回 406）

### 修复后（预期）：
- ✅ 服务器启动成功
- ✅ 无 406 错误（或者很少）
- ✅ 连接耗时：~2-5 秒（如果服务器已运行，~200-500ms）
- ✅ 检查请求：更少，且都成功（200/400/500）

## 连接慢的原因分析

### 主要原因：

1. **406 错误导致检查失败**
   - `checkJADXServerRunning` 返回 false（因为收到 406）
   - 代码认为服务器未运行，继续等待
   - 实际服务器已运行，但请求格式不对

2. **频繁的无效检查**
   - 每次检查都失败（406）
   - 需要等待多个检查周期才能最终成功
   - 浪费了大量时间

3. **服务器启动时间**
   - JADX MCP 服务器需要检查 JADX 插件连接（8650端口）
   - 这个过程需要 2-3 秒

### 修复后的流程：

1. **检查服务器**：使用正确的 Accept 头，立即检测到服务器运行状态
2. **如果已运行**：直接初始化连接（~200-500ms）
3. **如果未运行**：启动服务器，使用渐进式检查快速检测（~2-5秒）

## 总结

**核心问题：** `checkJADXServerRunning` 缺少 `Accept: application/json, text/event-stream` 请求头

**修复方案：**
1. ✅ 添加 Accept 请求头
2. ✅ 优化检查间隔（渐进式）
3. ✅ 改进日志输出（显示检查次数和耗时）

**预期效果：**
- 如果服务器已运行，连接时间从 20-30秒 减少到 200-500ms
- 消除大量 406 错误
- 减少无效的检查请求
