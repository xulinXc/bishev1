## 0. 方法与原则

1. 先把“现象”量化：失败数量、耗时、误报数量、复现条件。
2. 再把“链路”画清楚：客户端/服务端/插件/外部工具分别负责什么，错误在哪一段发生。
3. 修复优先级：高误报 > 解析失败 > 体验问题 > 性能优化。
4. 每个修复都要有“验证方式”：最小复现、对比数据、回归测试。

## 1. POC 扫描能力改善

### 1.1 批量解析失败 → 扩展解析兼容面

**问题现象**
- 内置 POC 库接入后，出现大量 YAML POC 解析失败，影响可用覆盖面。

**定位与根因**
- 失败文件集中在 Nuclei 变体与缩进/字段不规范的 X-Ray POC。
- 解析结构体字段覆盖不完整、解析逻辑过“严格”导致跳过可用文件。

**改进动作**
- 扩展 Nuclei 结构体字段，兼容 `url/redirect/detections/matchers` 等常见变体。
- 对 X-Ray POC：在 YAML 解析失败或结构异常时，从原始文本中做兜底提取（rules/expression）。
- Headers 解析支持多种形态（map / map[interface{}]interface{} / 数组形式的 `Key: Value`）。

**验证方式与效果**
- 批量统计：解析成功率提升到 80%+，无法解析的文件被自动跳过，不阻断扫描任务。
- 现状与后续方向见 [POC解析状态报告.md](../shili/说明/POC解析状态报告.md)。

### 1.2 误报率过高 → 收紧表达式评估与匹配条件

**问题现象**
- 扫描结果“命中很多”，但逐个验证不可利用，误报严重。

**定位与根因**
- 表达式评估存在“疑似逻辑/空响应也能通过”的路径。
- Nuclei POC 对 `matchers/detections` 的存在性要求不足，导致无约束模板也被当作有效命中。
- 布尔表达式对空表达式、AND 条件处理不严谨。

**改进动作**
- 对空响应、空 pattern 的 `bcontains/bmatches` 做严格判定。
- 移除或禁用“疑似命中即认为成功”的逻辑，改为严格满足全局表达式才报告漏洞。
- 统一并强化布尔表达式求值规则（空表达式为 false，AND 必须全 true）。
- 对 Nuclei：无 matchers/detections 的模板不接受，避免“天然真”。

**验证方式**
- 选取高误报场景（靶场/真实目标）做前后对比：命中数量显著下降，但有效命中更可信。
- 详细修复点与代码位置见 [POC扫描问题解决经验报告.md](../shili/说明/POC扫描问题解决经验报告.md)。

### 1.3 OOB 依赖的模板 → 明确不支持并避免误报

**问题现象**
- 部分 CVE 模板依赖 OOB（出网/回连）验证，本项目不具备该验证链路时会出现误判空间。

**改进动作**
- 在表达式评估中识别 OOB 相关关键字，统一返回 false，避免误报。

**代价与取舍**
- 这类漏洞将“不检测/不确认”，但总体扫描结果可信度提升。

## 2. MCP（JADX / IDA）集成改善

### 2.1 JADX MCP 502 → 区分“服务器在线”与“插件在线”

**问题现象**
- Go 侧访问 MCP 服务器正常（HTTP 200），但调用工具时返回 502。

**定位与根因**
- 错误链路发生在：JADX MCP 服务器尝试连接 JADX GUI 内部插件（端口 8650）失败。
- 这不是 Go 客户端的协议/请求问题，而是“插件未启动/GUI 未运行/端口不可达”。

**改进动作**
- 在工具调用返回中识别 502，给出明确提示与排查清单（GUI 是否运行、插件是否启动、端口是否监听）。
- 在可恢复场景下增加重连与重试（对 MCP 服务器侧异常有效，对 GUI 内插件离线仅能提示用户处理）。

**参考原始记录**
- 见 [JADX_MCP_502错误问题总结.md](../shili/ai/JADX_MCP_502错误问题总结.md)。

### 2.2 JADX MCP 406 与连接慢 → 修正请求头 + 降噪轮询

**问题现象**
- 启动检测阶段频繁出现 406，且连接耗时明显偏长。

**定位与根因**
- FastMCP streamable-http 场景要求特定 `Accept` 头；缺失时返回 406。
- 启动检测轮询过密，导致大量无效请求与日志噪音。

**改进动作**
- 为检测请求补齐 Accept 头，避免 406。
- 调整检测间隔策略（启动初期更快、后期退避），降低噪音与资源消耗。

**参考原始记录**
- 见 [JADX_406错误和连接慢问题修复.md](../shili/ai/JADX_406错误和连接慢问题修复.md)。

### 2.3 IDA MCP 协议误解 → 从“端点/传输模式”回到可用链路

**问题现象**
- 访问 `/mcp` 404、对 `/sse` POST 405、或方法不存在等协议不匹配现象。

**定位与根因**
- IDA MCP 相关组件存在“代理服务器（SSE transport）”与“插件端点（标准 /mcp）”两套模式。
- 如果把 SSE transport 当成标准 MCP POST 端点，会出现 404/405/方法不支持等错误。

**改进动作**
- 明确区分 transport 类型与实际可调用端点，选择可用的标准 JSON-RPC 调用链路。

**参考原始记录**
- 见 [IDA_MCP问题完整总结.md](../shili/ai/IDA_MCP问题完整总结.md)。

## 3. 工程可维护性与仓库卫生

### 3.1 生成物清理与忽略规则

**问题现象**
- 上传产物、工具缓存、虚拟环境等混入仓库，影响体积与可维护性。

**改进动作**
- 清理明显的运行生成物（上传目录、日志、缓存、虚拟环境等）。
- 补充根目录 `.gitignore`，避免生成物回流。

### 3.2 文档结构收敛

**问题现象**
- “论文资料 / 调试记录 / 使用说明”混在一起，重复命名文档造成检索困难。

**改进动作**
- 增加统一索引：见 [docs/README.md](./README.md)。
- 对重复命名的文档进行改名/归档，保留一份“主版本”，避免后续误引用。
