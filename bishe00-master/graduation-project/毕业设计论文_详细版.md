## 1. 项目背景与核心目标

### 1.1 项目背景

随着计算机硬件及软件的不断发展，互联网技术已经得到全世界人们的认可，在我们日常生活中的地位也是非常重要的，并渐渐融入了我们的生活，成为生活中不可缺少的一部分。企业业务快速向 Web 应用与移动程序迁移，网络安全威胁呈现多样化、复杂化趋势，公开端口暴露、薄弱鉴权接口、第三方组件漏洞与配置缺陷等安全风险集中涌现。在攻防演练、安全评估与渗透测试场景中，安全人员需要在有限时间内完成"资产摸排—脆弱性验证—证据收集"的完整闭环，同时满足等保2.0、数据与隐私合规对测试边界、影响控制与可审计性的严格要求。

### 1.2 现有工具的问题

然而，现有安全工具生态存在明显的碎片化问题：

1. **工具分散**：端口扫描工具（masscan/nmap）与目录枚举工具（ffuf/dirsearch）在各自领域高效，但与业务语义衔接有限，难以形成统一的验证与报告流程。

2. **规则不统一**：漏洞验证工具 Nuclei 与 X-Ray 规则体系成熟、漏洞覆盖全面，但在多步状态机编排与统一任务管理上成本较高，规则格式互不兼容导致维护负担重。

3. **数据不闭环**：JS 解包工具（packer-fuzzer）、URL 抽取工具（urlfinder）、Android 反编译工具（JADX）、二进制分析工具（IDA Pro）等单点工具丰富，但输入输出分散、缺少统一任务与数据流，人工搬运易丢信息、难沉淀可复用知识。

4. **AI 集成不足**：AI 驱动的安全分析能力逐步成熟，但缺乏与扫描工具链的深度集成，难以实现"发现→分析→验证→报告"的自动化闭环。

### 1.3 核心目标

本项目的核心目标是设计并实现一个**轻量一体化安全扫描平台 NeonScan**，解决工具碎片化、数据不闭环与合规可审计不足的问题。平台将端口扫描、目录枚举、POC/EXP 验证、Web 指纹探测、WAF 绕过与 JS/API 资产收集纳入统一后端与前端，采用统一的"任务+SSE 事件流"模型实现实时反馈。

### 1.4 主要功能目标

1. **端口扫描**：支持 TCP/UDP 协议、自定义端口范围、并发与超时控制、可选 Banner 抓取
2. **目录扫描**：多字典并发与超时控制、判定有效状态码 200/301/302/403、记录响应长度与 Location
3. **POC 扫描**：兼容 Legacy、X-Ray 与 Nuclei 子集三种 POC 规则格式
4. **EXP 执行**：多步骤编排、变量提取与替换、Cookie 贯穿、重试与休眠机制
5. **Web 探测**：标题采集、协议识别、指纹特征识别
6. **WAF 绕过**：多种绕过策略生成与验证
7. **JS 资产收集**：可解包性判定、JS 解包、ffuf 字典爆破、urlfinder URL 抓取
8. **MCP 集成**：IDA Pro MCP 集成、JADX MCP 集成
9. **AI 分析**：AI 网关设计、自动化扫描工作流、智能报告分析

## 2. 项目解决的实际问题与现实意义

### 2.1 解决的实际问题

#### 2.1.1 工具碎片化问题

传统安全测试需要使用多个独立工具：端口扫描用 nmap/masscan，目录扫描用 ffuf/dirsearch，POC 验证用 Nuclei/X-Ray，JS 分析用 packer-fuzzer。这种分散的工具链带来以下问题：

- **学习成本高**：每个工具都有独特的使用方式和参数格式
- **工作流程割裂**：需要在多个工具之间手动搬运数据
- **结果分散**：难以形成统一的报告和视图

**NeonScan 的解决方案**：将所有功能集成到统一平台，使用统一的任务管理和数据格式，用户只需学习一个工具即可完成全流程安全测试。

#### 2.1.2 数据不闭环问题

传统工具链中，各个环节产生的数据无法自动流转：

- JS 资产收集发现的 URL 不会自动导入目录扫描
- 目录扫描发现的接口不会自动触发 POC 验证
- POC 验证结果不会自动生成可执行的 EXP

**NeonScan 的解决方案**：通过数据流闭环设计，将资产发现结果自动回流至漏洞验证环节。JS/API 清单可一键导入目录扫描、POC 扫描模块，形成"资产发现→验证→报告"的完整数据流闭环。

#### 2.1.3 合规可审计不足问题

在等保2.0和数据合规要求下，安全测试需要：

- 记录所有操作以便审计
- 控制测试影响范围
- 自动清理敏感数据

**NeonScan 的解决方案**：采用目录隔离与定期清理机制，文件存储至 `uploads/u-时间戳/` 目录，支持7天异步清理，满足合规要求。

### 2.2 现实意义

1. **降低安全测试门槛**：一体化平台让非专业安全人员也能进行基本的安全评估
2. **提高测试效率**：自动化工作流减少人工操作，平均总耗时降低 ≥40%
3. **提升发现率**：数据闭环设计使目录命中率提升 ≥30%
4. **满足合规要求**：完整的审计日志和数据清理机制符合等保2.0要求

### 3.2 选择这种架构的理由

#### 3.2.1 B/S 架构的优势

| 对比项 | B/S 架构 | C/S 架构 | CLI 工具 |
|--------|----------|----------|----------|
| 跨平台 | ✅ 浏览器即可 | 需要安装客户端 | ✅ 跨平台但需配置环境 |
| 无需安装 | ✅ | 需要安装 | ✅ 但需配置依赖 |
| 易于维护 | ✅ 更新服务器即可 | 需要逐个更新客户端 | 无需维护 |
| 界面友好 | ✅ 可视化界面 | ✅ 可视化界面 | ❌ 需要记忆命令 |
| 适合专业工具 | ❌ | ✅ | ✅ |

**选择理由**：
- 目标用户需要可视化界面降低学习成本
- 无需安装的特性便于快速部署和使用
- 统一平台避免多工具切换

#### 3.2.2 Go 语言的优势

1. **强大的并发处理能力**：通过 goroutine 和 channel 实现轻量级协程，能够轻松处理高并发的网络请求和 I/O 操作

2. **优秀的网络编程能力**：标准库 `net/http` 提供了完整的 HTTP 服务器和客户端实现，支持 SSE 事件流、TLS 加密、连接池管理等高级特性

3. **内存安全**：严格的类型系统和内存安全机制，能够有效防止缓冲区溢出、空指针引用等常见安全问题

4. **静态编译**：能够生成单一可执行文件，无需运行时环境，降低了部署成本和安全风险

5. **快速编译**：编译速度非常快，能够在几秒钟内编译大型项目

### 3.3 核心设计模式

#### 3.3.1 任务管理模型

系统采用统一的"任务 + SSE 事件流"模型：

```go
type Task struct {
    ID        string          // 唯一任务ID
    Total     int             // 任务总数
    Done      int             // 已完成数量
    Created   time.Time       // 创建时间
    ch        chan SSEMessage // SSE消息通道
    stop      chan struct{}   // 停止信号
}
```

所有长任务都通过任务管理机制进行统一管理，用户启动扫描任务后，系统创建任务对象，分配唯一任务 ID，通过 SSE 连接实时推送任务进度和结果。

#### 3.3.2 数据流闭环设计

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   资产发现    │ ──▶ │   漏洞验证    │ ──▶ │   报告生成    │
│              │     │              │     │              │
│ - 端口扫描   │     │ - POC扫描    │     │ - AI分析    │
│ - 目录扫描   │     │ - EXP执行    │     │ - 结构化报告 │
│ - JS资产收集 │     │ - WAF绕过    │     │ - 修复建议   │
└──────────────┘     └──────────────┘     └──────────────┘
        │                   │                    │
        └───────────────────┴────────────────────┘
                    数据回流
```

JS/API 清单可一键导入目录扫描、POC 扫描与 Web 探测模块，减少盲测、提升命中率。

## 5. 核心算法与逻辑实现

### 5.1 端口扫描算法

**核心逻辑**：

```
1. 解析端口范围字符串（如 "1-1000,3000-4000"）
2. 创建任务对象，分配唯一任务ID
3. 使用信号量（semaphore）控制并发数
4. 启动 goroutine 并发探测每个端口
5. TCP：net.DialTimeout 建立连接
6. UDP：net.DialUDP 发送数据包
7. 可选：读取 Banner 获取服务版本
8. 通过 SSE 实时推送进度和结果
9. 全部完成后执行清理
```

**关键代码**：

```go
func portScanHandler(w http.ResponseWriter, r *http.Request) {
    ports := parsePorts(req.Ports)  // 解析端口范围
    t := newTask(len(ports))        // 创建任务
    
    go func() {
        sem := make(chan struct{}, concurrency)  // 信号量控制并发
        var wg sync.WaitGroup
        
        for _, p := range ports {
            wg.Add(1)
            sem <- struct{}{}  // 获取信号量
            
            go func() {
                defer wg.Done()
                addr := fmt.Sprintf("%s:%d", host, p)
                
                // TCP 扫描
                conn, err := net.DialTimeout("tcp", addr, timeout)
                if err == nil {
                    // 读取 Banner
                    if grabBanner {
                        buf := make([]byte, 256)
                        n, _ := conn.Read(buf)
                        banner := string(buf[:n])
                        sendResult(t, p, "open", banner)
                    }
                    conn.Close()
                }
                
                <-sem  // 释放信号量
            }()
        }
        wg.Wait()
        finishTask(t.ID)
    }()
}
```

### 5.2 目录扫描算法

**核心逻辑**：

```
1. 加载字典文件（支持多字典合并）
2. 使用工作池模式控制并发
3. 发送 HTTP 请求到目标路径
4. 判定有效状态码（200/301/302/403）
5. 记录响应长度和 Location
6. 通过 SSE 实时推送发现的目录
```

### 5.3 POC 扫描算法

**支持的规则格式**：

#### Legacy 格式

```yaml
name: "ThinkPHP 5.0.23 RCE"
method: POST
path: "/index.php?s=index/think\\app/invokefunction"
body: "vars[0]=system&vars[1][]=whoami"
validate:
  status: [200]
  body_contains: "root"
```

#### X-Ray 格式

```yaml
id: thinkphp-5023-rce
info:
  name: ThinkPHP 5.0.23 RCE
rules:
  method: POST
  path: "/index.php"
  body: "s=index/think\\app/invokefunction"
  expression: "response.status == 200 && response.body.bcontains(b'root')"
```

#### Nuclei 格式

```yaml
id: thinkphp-5023-rce
info:
  name: ThinkPHP 5.0.23 RCE
requests:
  - method: POST
    path: "/index.php"
    body: "s=index/think\\app/invokefunction"
matchers:
  - type: word
    words:
      - "root"
```

### 5.4 EXP 执行算法

**核心逻辑**：

```
1. 解析 EXP 文件中的多步骤定义
2. 按顺序执行每个步骤：
   a. 变量替换（如 {{BaseURL}}, {{randstr}}）
   b. 发送 HTTP 请求
   c. 从响应中提取变量（BodyRegex/HeaderRegex）
   d. Cookie 管理（贯穿整个利用链）
3. 支持重试和休眠机制
4. 验证每个步骤的命中条件
5. 返回执行结果
```

### 5.5 AI 驱动的自动化工作流

**核心逻辑**：

```
1. 用户配置 AI 提供商（DeepSeek/OpenAI/Ollama）
2. 启动自动化扫描：
   a. 端口扫描 → 获取开放端口
   b. Web 探测 → 获取服务指纹
   c. 目录扫描 → 发现敏感路径
   d. POC 扫描 → 验证漏洞
3. AI 分析扫描结果：
   a. 识别漏洞类型和风险等级
   b. 生成漏洞利用建议
   c. 自动生成 EXP 代码（如需要）
4. AI 生成结构化报告
```

## 6. 技术难点与解决方案

### 6.1 SSE 长连接稳定性

**问题**：SSE 长连接在长时间运行或网络不稳定时可能断开。

**解决方案**：
- 浏览器原生支持 EventSource API，自动重连
- 服务器实现心跳机制，定期发送注释行保持连接
- 任务完成后主动关闭连接，避免资源泄漏

### 6.2 高并发端口扫描

**问题**：大量端口同时扫描会耗尽系统资源或触发目标防护。

**解决方案**：
- 使用信号量（semaphore）模式控制并发数
- 可配置的超时时间，避免连接等待
- UDP 扫描使用单独的并发控制

### 6.3 POC 规则格式兼容

**问题**：三种规则格式语法不同，解析逻辑复杂。

**解决方案**：
- 设计统一的中间表示（IR）
- 每种格式转换为统一的 IR 再执行
- 支持表达式引擎处理复杂匹配逻辑

### 6.4 AI EXP 生成与验证

**问题**：AI 生成的 EXP 代码可能存在语法错误或逻辑问题。

**解决方案**：
- 自动验证生成的 EXP
- 检测常见错误（语法、缩进等）
- AI 分析失败原因并自动修正
- 最多 5 次重试迭代

### 6.5 外部工具集成

**问题**：ffuf、urlfinder、JADX 等外部工具使用方式各异。

**解决方案**：
- 命令模板化封装
- 分阶段回传 stdout/stderr
- 统一结果格式输出

### 6.6 文件上传安全

**问题**：用户可能上传恶意文件或超大文件。

**解决方案**：
- 单文件大小限制（≤512MB）
- 文件存储至隔离目录（`uploads/u-时间戳/`）
- 7 天异步清理机制

## 附录 A：常见答辩问题

### Q1: 请简要介绍你的毕业设计项目背景和核心目标

答：本项目设计并实现了一个名为 NeonScan 的一体化安全扫描平台。随着企业业务向 Web 应用迁移，网络安全威胁日益复杂，传统安全工具存在碎片化、数据不闭环的问题。本项目的核心目标是整合端口扫描、目录枚举、POC 验证、EXP 执行等安全测试功能，通过统一的平台和"任务+SSE 事件流"模型，实现"资产发现→验证→报告"的完整闭环，提高安全测试效率和覆盖率。

### Q2: 项目解决了什么实际问题？有什么现实意义？

答：本项目解决了三个实际问题：1）工具碎片化问题，传统安全测试需要使用多个独立工具，学习成本高；2）数据不闭环问题，各个环节产生的数据无法自动流转；3）合规可审计不足问题。现实意义在于：降低安全测试门槛、提高测试效率（平均总耗时降低≥40%）、满足等保2.0合规要求。

### Q3: 项目的技术架构是如何设计的？为什么选择这种架构？

答：项目采用 B/S 架构，前端使用原生 HTML/CSS/JavaScript，后端使用 Go 语言单进程 HTTP 服务。选择理由：1）B/S 架构适合需要可视化界面的安全工具，用户无需安装即可使用；2）Go 语言具有强大的并发处理能力和优秀的网络编程能力，非常适合高并发扫描场景；3）静态编译特性使部署变得简单，生成单一可执行文件无需依赖环境。

### Q4: 你使用了哪些关键技术或框架？选择它们的理由是什么？

答：主要技术包括：1）Go 语言 - 高并发、低延迟、静态编译；2）SSE 事件流 - 实现服务器向客户端实时推送扫描进度；3）MCP 协议 - 实现与 JADX 等外部工具的标准化集成（注：IDA MCP 在后续版本规划中）；4）多种 POC 规则格式兼容（Legacy/X-Ray/Nuclei）- 降低用户学习成本，复用现有规则生态。

### Q5: 项目的核心算法或逻辑是如何实现的？

答：核心逻辑包括：1）端口扫描使用 goroutine+信号量模式实现高并发探测；2）目录扫描采用工作池模式控制并发，判定有效状态码；3）POC 扫描通过统一的表达式引擎兼容三种规则格式；4）AI 驱动的自动化工作流通过 AI 分析扫描结果，自动生成下一步扫描建议和修复方案。

### Q6: 你在项目中遇到的最大技术难点是什么？如何解决的？

答：最大的难点是 AI EXP 生成与验证。AI 生成的代码可能存在语法错误或逻辑问题。解决方案：1）设计自动验证机制，在目标环境实际执行 EXP；2）检测常见错误（语法、缩进、输出提取等）；3）AI 分析失败原因并自动修正；4）最多 5 次重试迭代，确保生成的 EXP 可用。

### Q7: 项目的创新点体现在哪些方面？

答：创新点包括：1）一体化平台设计，整合分散的安全工具；2）数据流闭环，资产发现结果自动回流至漏洞验证环节；3）多格式 POC 兼容，降低用户学习成本；4）AI 驱动的工作流，实现智能化分析和自动化决策；5）MCP 协议集成专业逆向工具，扩展系统能力。

### Q8: 系统的主要功能模块有哪些？它们之间如何协作？

答：主要模块包括：端口扫描、目录扫描、POC 扫描、EXP 执行、Web 探测、WAF 绕过、JS 资产收集、MCP 集成、AI 分析。协作方式：采用统一的"任务+SSE 事件流"模型，JS/API 清单可一键导入其他扫描模块，形成数据流闭环。例如：端口扫描发现端口 → Web 探测识别技术栈 → 目录扫描发现敏感路径 → POC 扫描验证漏洞 → AI 分析生成报告。

### Q9: 你是如何进行系统测试的？测试结果如何？

### Q11: 系统是否存在安全风险？你做了哪些安全防护措施？

答：存在的安全风险：文件上传可能包含恶意文件、端口扫描可能被滥用、扫描结果包含敏感信息。防护措施：1）单文件大小限制（≤512MB）；2）文件存储至隔离目录；3）7天自动清理机制；4）API 密钥仅存内存不落盘；5）仅支持本地部署，明确免责声明。

### Q12: 项目的数据来源是什么？数据是如何处理和存储的？

答：数据来源：用户输入（目标信息、扫描参数）、实时扫描结果、用户上传的 POC/字典/EXP 文件。数据处理：用户输入→解析验证→任务创建→并发执行→结果收集→SSE推送。存储：扫描进度和结果保存在内存的 Task 对象中，用户上传文件存储在 `uploads/u-时间戳/` 目录，扫描报告保存在客户端 localStorage。

```go
func sseHandler(w http.ResponseWriter, r *http.Request) {
    id := r.URL.Query().Get("task")
    t, ok := getTask(id)
    if !ok {
        http.Error(w, "task not found", http.StatusNotFound)
        return
    }
    
    w.Header().Set("Content-Type", "text/event-stream")
    w.Header().Set("Cache-Control", "no-cache")
    w.Header().Set("Connection", "keep-alive")
    
    for {
        select {
        case <-r.Context().Done():
            return
        case msg, ok := <-t.ch:
            if !ok {
                return
            }
            writeSSE(w, msg)
        }
    }
}
```
