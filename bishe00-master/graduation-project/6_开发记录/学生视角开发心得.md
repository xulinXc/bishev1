# 开发心得：从学生视角聊聊这个项目

## 前言

作为一个计算机专业的学生，第一次独立完成一个比较完整的项目，还是有不少话想说的。这个安全扫描平台从零开始做到现在，虽然还有很多可以优化的地方，但确实踩了不少坑，也学到很多东西。今天就从一个过来人的角度，聊聊开发过程中遇到的那些事儿。

---

## 一、为什么做这个项目

说实话，一开始只是觉得网络安全挺有意思的，特别是渗透测试那种"发现漏洞-利用-获得权限"的过程很有成就感。但是真正开始做的时候才发现，网上那些工具虽然功能强大，但都很分散：端口扫描用 nmap，目录扫描用 ffuf，POC 验证用 Nuclei，每次做安全测试都要在好几个工具之间来回切换，数据格式也不统一，整理报告的时候特别头疼。

后来看到一些商业的安全扫描平台，一体化的设计感觉很方便，就想着能不能自己做一个。跟导师沟通后，导师也觉得这个方向可行，一体化安全扫描平台既有实用价值，技术难度也适中，适合作为毕业设计。

---

## 二、遇到的问题和解决办法

### 1. Go语言是第一次用

**问题**：之前做项目都是用 Python，Go语言完全没接触过。一开始连基本的语法都不熟悉，更别说用它写 Web 服务了。

**解决**：
- 花了大概两周时间过了一遍《Go语言圣经》，重点看了 goroutine、channel、接口这些核心概念
- 跟着教程写了几个小例子，比如 HTTP 服务器、文件操作之类的
- 开始正式写项目的时候，遇到不会的就查官方文档，Go 的文档写得特别清晰
- 不会的就在 Stack Overflow 和 GitHub 上搜，Go 社区的解决方案挺多的

**感受**：Go 的学习曲线其实没有想象的那么陡，语法比 C++ 简单很多，关键是理解它的并发模型。一旦搞懂了 goroutine 和 channel，用起来就很顺手了。

---

### 2. 不知道怎么让前端和后端交互

**问题**：后端写好了接口，但不知道前端怎么调用，也不知道怎么实时显示扫描进度。最早想的是前端轮询，每隔几秒请求一次接口，但这样既浪费资源，实时性也差。

**解决**：
- 后来了解到了 SSE（Server-Sent Events）这个技术，服务器可以主动向浏览器推送数据
- 前端用原生的 EventSource API，浏览器会自动处理重连
- 后端维护一个任务列表，每个任务有自己的 channel，扫描进度实时写入 channel，SSE handler 再推送给前端

**关键代码**：

```go
// 后端：维护任务和channel的映射
var tasks = make(map[string]*Task)

type Task struct {
    ID    string
    Total int
    Done  int
    ch    chan SSEMessage  // 这个channel用来传递消息
    stop  chan struct{}
}

// 前端：原生EventSource，不需要任何库
const eventSource = new EventSource(`/sse?task=${taskId}`);
eventSource.onmessage = (event) => {
    const data = JSON.parse(event.data);
    // 更新进度条、显示结果
};
```

**感受**：SSE 比 WebSocket 简单太多了，不需要前端库，服务器端也不需要特殊的库，Go 标准库直接支持。最重要的是它适合"服务器推数据，客户端展示"的场景，正好符合扫描进度的需求。

---

### 3. POC 规则格式不统一

**问题**：网上有各种格式的 POC，Legacy 的、X-Ray 的、Nuclei 的，格式都不一样。一开始只支持一种，后来发现很多现成的 POC 用不了，还要自己改格式，很麻烦。

**解决**：
- 花了时间研究了这三种格式的区别和共性
- 设计了一个统一的中间表示，不管什么格式都转成这个中间格式再执行
- 支持表达式匹配，这样 X-Ray 的那种复杂逻辑也能处理

**三种格式对比**：

| 格式 | 特点 | 我怎么支持 |
|------|------|-----------|
| Legacy | 最简单，就是发请求看响应 | 最基础的支持 |
| X-Ray | 可以写表达式，如 `status==200 && body.contains("test")` | 用表达式引擎解析 |
| Nuclei | 社区最活跃，规则最多 | 解析 matchers 部分 |

**感受**：这个花了比较多的时间研究，但做完后觉得值了。现在可以直接用网上现成的 Nuclei 规则库，不用自己一条条写，节省了大量时间。

---

### 4. AI 生成代码的坑

**问题**：想让 AI 自动生成 EXP 代码，但是 AI 第一次生成的代码经常有各种问题：语法错误、缩进不对、输出格式不对等等。直接用根本跑不通。

**解决**：
- 不能完全依赖 AI，自己要做校验
- 先生成代码，然后在本地跑一下，捕获错误
- 根据错误类型让 AI 修正，比如语法错误就让它修复缩进和引号
- 有些常见错误我写了自动修复脚本，比如添加输出标记、修正正则表达式
- 最多次尝试 5 次，超过就放弃，让用户知道这个方法行不通

**流程**：

```
AI生成代码 → 保存到文件 → Python执行 → 捕获输出
    ↓
有错误？ → 分析原因 → 让AI修正 → 再试
    ↓
失败5次 → 放弃，提示用户
```

**感受**：AI 是个好工具，但不能完全相信它。一定要有验证机制，不然生成的代码很可能跑不通。这个经验教训很宝贵，以后用 AI 辅助编程都要记得加校验。

---

### 5. 不知道怎么集成 IDA 和 JADX

**问题**：想做二进制分析和 APK 反编译，但 IDA 和 JADX 都是独立的桌面程序，不知道怎么跟 Web 系统集成。总不能要求用户同时开着 IDA 和浏览器吧。

**解决**：
- 发现了 MCP（Model Context Protocol）这个协议
- 查了 IDA 和 JADX 都有 MCP 服务器的解决方案
- 通过 HTTP/JSON-RPC 跟 MCP 服务器通信
- Web 系统只需要发请求，就能调用 IDA/JADX 的功能，结果还能实时返回

**架构**：

```
浏览器 ← HTTP → Go后端 ← JSON-RPC → IDA/JADX MCP服务器 ← → IDA/JADX本体
```

**感受**：MCP 这个协议真的很巧妙，解耦了 AI 工具和具体实现。虽然一开始配置有点麻烦，但完成后整个流程很顺畅，用户体验也很好。

---

### 6. 前端页面不好看

**问题**：后端功能都写好了，但前端页面丑得没法看。自己又不是设计专业出身，CSS 写得乱七八糟，配色也不协调。

**解决**：
- 借鉴了一些开源的安全工具界面，比如 Burp Suite、OWASP ZAP
- 使用了深色主题，本来安全工具就流行这个
- 渐变色和圆角用上，看起来现代一点
- 参考了一些 UI 设计的配色方案

**CSS 小技巧**：

```css
/* 深色主题 */
body {
    background: rgba(20, 20, 30, 0.63);
    color: white;
}

/* 渐变背景 */
button {
    background: linear-gradient(135deg, #667eea 0%, #764ba2 100%);
}

/* 圆角和阴影 */
.card {
    border-radius: 12px;
    box-shadow: 0 4px 6px rgba(0, 0, 0, 0.1);
}
```

**感受**：前端设计确实是个技术活，不是简单地把功能堆上去就行。好在这个项目不需要特别复杂的设计，主要精力还是放在功能实现上。

---

### 7. 文件上传的安全问题

**问题**：用户可以上传 POC 文件和字典文件，万一有人上传恶意文件怎么办？还有上传太大文件把服务器撑爆怎么办？

**解决**：
- 限制单个文件大小，最大 512MB
- 上传的文件存在单独的目录里，不跟系统文件混在一起
- 目录名用时间戳命名，u-20260404120000 这种格式
- 写了个定时任务，每天清理一次 7 天前的上传文件

**代码**：

```go
// 检查文件大小
if fileSize > 512*1024*1024 {
    http.Error(w, "文件太大", http.StatusBadRequest)
    return
}

// 隔离存储
uploadDir := fmt.Sprintf("uploads/u-%d/", time.Now().UnixNano())
os.MkdirAll(uploadDir, 0755)
```

**感受**：安全问题真的是一环扣一环，写代码的时候就要考虑到，不能等出事了再补救。

## 三、为什么选择这些技术

### 1. Go 语言

一开始考虑过 Python，但最后还是选了 Go，主要原因：

**性能好**：安全扫描需要并发探测大量端口，Go 的并发性能比 Python 好很多。同样是扫描 1000 个端口，Go 可能 10 秒，Python 要 30 秒以上。

**部署简单**：Go 可以编译成单个二进制文件，直接拷贝到服务器就能跑。Python 要装依赖，万一服务器环境不一样还可能出问题。

**内存占用低**：Python 跑久了内存会越来越高，Go 的内存管理好很多，垃圾回收也高效。

**实战经验**：虽然之前没正式用过 Go，但学过 C++ 的底子在那里，学起来不算太难。关键是看中了它的并发性能，这个对扫描工具太重要了。

---

### 2. 原生 HTML/CSS/JS

没用 Vue、React 这些框架的原因：

**简单直接**：项目不大，没必要引入一整套前端框架。
**零依赖**：不需要 npm、webpack 这些构建工具，文件直接在服务器上放着，浏览器原生支持。
**好维护**：代码都在眼前，没有额外的抽象层，新功能加上去很容易。

**对于安全工具来说**：界面不需要多复杂，能显示结果、能交互就行。过度设计反而增加复杂度。

---

### 3. SSE 而不是 WebSocket

**选择 SSE 的理由**：

- 单向通信就够了：服务器推扫描进度，浏览器只管接收和显示
- 实现简单：几行代码就能搭起来
- 浏览器原生支持：不需要引入 socket.io 这些库
- 自动重连：浏览器会自己处理网络断开的情况

WebSocket 当然也能用，但有点杀鸡用牛刀了。

---

### 4. 多种 POC 格式支持

**原因**：不想重复造轮子。

Nuclei 的规则库特别丰富，网上有几千条各种漏洞的规则。如果只支持自己定义的格式，这些现成的规则都用不了。

支持三种格式后，基本上网上常见的 POC 都能直接拿来用，省了大量时间。

---

## 四、实际用到的技术优势

### 1. Go 的并发模型

```go
// 控制并发数
sem := make(chan struct{}, 50)  // 最多50个并发

for _, port := range ports {
    sem <- struct{}{}  // 获取信号量
    go func() {
        defer func() { <-sem }()  // 释放信号量
        // 扫描逻辑
    }()
}
```

这个模式很好用，比起 Python 的 threading，Go 的 goroutine 轻量得多，开几千个都没问题。

### 2. 任务管理系统

所有扫描任务统一管理，任务 ID 一直有效，页面刷新也不会丢。任务状态都存在内存里，有专门的清理机制防止内存泄漏。

### 3. 数据闭环设计

这个是后来加的，发现确实好用。JS 扫描出来的 URL 直接就能导入目录扫描，不需要手动复制粘贴。结果还能回流到 POC 扫描，形成完整的数据流。

## 五、不足和改进空间

说实话，这个项目还有很多可以改进的地方：

**性能方面**：
- 还没做成分布式的，单机性能有上限
- 大量目标扫描的时候可能会卡
- 可以考虑用 Redis 做任务队列，支持多节点

**功能方面**：
- 还没有用户系统，多人共用一台服务器会混乱
- 报告只能导出 Markdown，能不能加个 PDF
- POC 规则解析还有 bug，极少数特殊格式解析不对

**代码方面**：
- 有些地方的错误处理不够完善
- 注释写得不够详细，后来自己看都有些费劲
- 单元测试覆盖率太低，很多边界情况没测到

这些问题留着以后慢慢优化吧，毕竟毕业设计时间有限，先把核心功能做出来。

---

## 结语

做这个项目最大的感触就是：**技术只是工具，思路才是关键**。很多时候遇到问题，不是找不到技术方案，而是不知道怎么把问题拆解成可解决的小问题。

Go 语言也好，SSE 也好，MCP 也好，都是实现想法的手段。真正有价值的是：**发现工具碎片化的痛点，提出一体化解决的思路，然后用合适的技术把它实现出来**。

希望这个项目以后还能继续维护下去，毕竟安全这行总是在不断出现新的漏洞和攻击方式，工具也要跟着进化。

---

**写给自己**：答辩的时候别紧张，把项目的来龙去脉讲清楚就行。老师问的每个问题背后都有原因，问技术选型是想知道你对技术的理解程度，问难点是想看你解决问题的能力。把自己踩过的坑、总结的经验讲出来，比背概念要有用得多。
