# NeonScan 安全扫描平台 - 开发心得与经验总结

## 从学生视角聊聊这个项目：踩坑、解决与技术选择

---

### 前言

我是计算机专业的一名普通学生，在做这个毕业设计之前，编程经验主要来自于课程作业和几个练手的小项目。说实话，在确定这个课题之前，我甚至没有完整地开发过一个涉及前后端交互、并发编程、协议解析等多个技术领域的综合性项目。

这个安全扫描平台的开发，从选题、技术调研、框架搭建、功能实现到最后的优化调试，前前后后花了将近半年的时间。这期间踩了无数的坑，也总结出了一些经验。现在回过头来看，很多当时觉得难以逾越的问题，回顾起来其实都有解决的办法，只是当时经验不足走了弯路。

今天把开发过程中的遇到的问题、解决思路、技术选型理由等详细记录下来，既是对自己这段经历的总结，也希望能给后续做类似项目的同学一些参考。

**重要说明**：本文不是技术教程，不会过多介绍语法细节或者 API 怎么调用。我会重点讲述：在什么场景下遇到了什么问题，为什么这个问题难，解决的思路是什么，最终选择了什么方案，这个方案有什么坑或者后续优化点。

## 第一部分：项目的由来和背景

### 1.1 选题的思考过程

确定毕业设计选题之前，我在导师提供的几个方向之间犹豫了很久。一开始考虑过做一个简单的学生信息管理系统，因为之前做过类似的项目，上手快、风险低。但后来仔细想了想，这样做出来的项目在答辩的时候可能很难出彩，评委老师每天要看几十个学生项目，如果都是增删改查，很难留下印象。

后来注意到网络安全这个方向。。一方面是自己平时对安全技术比较感兴趣，平时也会关注一些漏洞新闻和渗透测试的技术文章；另一方面是这个领域有很多现成的工具可以做参考，不像其他领域那样需要从零开始设计功能。

在跟导师沟通的时候，导师提了一个问题：网上已经有那么多安全工具了，你做的跟它们有什么区别？这个问题让我思考了很久。

经过调研，我发现了现有安全工具的几个痛点：

第一个是工具太分散。做一次完整的安全测试，可能要用到 nmap 做端口扫描、用 ffuf 或者 dirsearch 做目录枚举、用 Nuclei 做 POC 验证、甚至还要用脚本来处理数据。每次切换工具都要重新配置参数，输出格式也不统一，整理报告的时候特别麻烦。

第二个是数据不闭环。目录扫描发现的接口不会自动触发 POC 验证，POC 验证结果也不会自动生成可执行的利用代码。整个工作流程是割裂的，需要人工不断重复"发现-验证-利用"的循环。

第三个是好用的工具都是收费的。商业的安全扫描平台功能确实强大，但价格对于个人或者小团队来说不太友好。而开源的工具往往功能单一，或者集成度不够高。

基于这些观察，我跟导师提出了做一个"一体化安全扫描平台"的设想。导师听完之后觉得方向可行，但提醒我不要贪多嚼不烂，要先确定好核心功能范围，把主要功能做稳定了，再考虑扩展。

### 1.2 确定核心功能范围

经过讨论，我们确定了平台的核心功能：

**必做功能（保证项目完整性）**：
- 端口扫描：TCP 和 UDP 两种协议，支持自定义端口范围
- 目录扫描：多字典并发扫描，状态码判定
- POC 扫描：兼容主流的规则格式
- 任务管理：实时进度显示，中途可停止

**进阶功能（体现项目亮点）**：
- EXP 执行：自动化漏洞利用
- Web 探测：指纹识别、技术栈检测
- WAF 绕过：多种 Payload 变形策略
- AI 分析：智能扫描工作流

**扩展功能（展示可扩展性）**：
- JS 资产收集：解包、URL 提取
- MCP 集成：JADX（注：IDA MCP 有调研但未最终实现）

最后实际上把这些功能都实现了，但开发过程确实是分阶段的。先把必做功能做稳，再逐步添加进阶功能，扩展功能是在后期时间比较充裕的情况下加的。

## 第二部分：技术挑战与解决方案

### 2.1 Go 语言入门：从零到能写项目

#### 2.1.1 入门阶段的困难

刚开始学习 Go 语言的时候，思维方式要从 Python 切换过来，有一段时间特别不适应。

Python 是解释型语言，写完直接跑，错误信息很清楚。Go 是编译型语言，每次改动都要重新编译，编译错误有时候莫名其妙，比如少了一个分号或者括号没对齐。而且 Go 的语法比 Python 严格很多，比如 if 条件不用括号但必须加大括号，函数返回值要提前声明类型等等。

更麻烦的是并发编程。Python 有 GIL，做多线程其实还是单核运行。Go 的 goroutine 是真正的多核并发，但我之前完全不理解"并发"和"并行"的区别，也不知道 channel 怎么用。看了几遍教程觉得懂了，真正写的时候又蒙了。

#### 2.1.2 学习路径和方法

我的 Go 语言学习大概分成了几个阶段：

**第一阶段（两周）：语法基础**

这个阶段主要过了一遍 Go 语言的基础语法，包括变量声明、函数定义、控制流、数组切片、map、结构体等。我用的是《Go语言圣经》这本书，配合 Go 官方文档的 Tour of Go 做练习。

这个阶段不需要死记硬背，只需要知道：
- 变量声明用 `var` 或者 `:=`
- 数组是值类型，切片是引用类型
- map 的操作方式跟 Python 字典差不多
- 结构体可以理解成没有方法的轻量级类

**第二阶段（两周）：接口和网络编程**

接口是 Go 的精髓，但也是最不好理解的部分。我花了比较长的时间理解接口的概念和用法。

接口，简单来说就是"行为"的抽象。只要一个类型实现了某个接口定义的所有方法，那它就属于这个接口类型，不需要显式声明"实现"关系。这跟 Java 的 implements 关键字、C++ 的虚函数表不一样，更像是 Python 的 duck typing。

网络编程部分主要是学习了 Go 标准库的 net/http 包。官方文档里有完整的 HTTP 服务器和客户端示例，跟着敲一遍就能理解基本用法。

**第三阶段（持续）：并发编程**

这是 Go 最有特色的地方，也是我花了最长时间才理解的部分。

goroutine 可以理解成轻量级的线程，创建成本很低，可以同时运行成千上万个。channel 是 goroutine 之间的通信通道，可以用来传递数据或者同步操作。

经典的例子是生产者和消费者模型：

```go
func producer(ch chan int) {
    for i := 0; i < 10; i++ {
        ch <- i  // 发送数据到 channel
    }
    close(ch)  // 关闭 channel
}

func consumer(ch chan int) {
    for v := range ch {  // 持续从 channel 读取数据，直到 channel 关闭
        fmt.Println(v)
    }
}

func main() {
    ch := make(chan int)  // 创建一个 int 类型的 channel
    go producer(ch)       // 启动生产者 goroutine
    consumer(ch)          // 主 goroutine 执行消费者
}
```

这个例子很好理解，但实际项目中怎么用，还是踩了不少坑。

#### 2.1.3 实际项目中的经验

在写端口扫描模块的时候，我遇到了一个典型的问题：如何控制并发数量。

一开始我以为 goroutine 是"无限多"的，就写成了这样：

```go
// 错误写法：会创建太多 goroutine
for _, port := range ports {
    go func() {
        // 扫描端口的逻辑
    }()
}
```

当端口数量很多的时候（比如扫描 65535 个端口），程序会瞬间创建 6 万多个 goroutine，导致内存爆炸或者系统资源耗尽。

正确做法是用信号量（Semaphore）模式控制并发数量：

```go
// 正确写法：控制并发数量
concurrency := 50  // 最多50个并发
sem := make(chan struct{}, concurrency)

for _, port := range ports {
    sem <- struct{}{}  // 获取信号量，如果满了会阻塞
    go func(port int) {
        defer func() { <-sem }()  // 函数结束的时候释放信号量
        // 扫描端口的逻辑
    }(port)
}
```

`make(chan struct{}, 50)` 创建了一个带缓冲区大小的 channel，缓冲区可以理解成"许可数量"。往 channel 里写数据会消耗一个许可，读数据会释放一个许可。当所有许可都被占用时，再往 channel 写数据会阻塞，直到有其他 goroutine 读取数据释放许可。

这个模式在 Go 社区叫做"Semphore"，是控制并发数的标准做法。

**经验总结**：
- Go 的编译错误信息虽然有时候不太直观，但定位比较准确，看到错误信息要仔细读
- 接口是隐式实现，不需要记住谁实现了谁，关键是理解"行为"抽象
- 并发编程要时刻想着"资源控制"，不要无限制地创建 goroutine
- 善用 `go fmt` 和 `go vet`，可以避免很多语法和常见错误

### 2.2 前端后端交互：SSE 实时通信

#### 2.2.1 为什么选择 SSE

在确定使用 SSE 之前，我调研过几种不同的方案。

**第一种是轮询（Polling）**。前端每隔几秒请求一次接口，获取最新进度。这是最简单的实现方式，代码写起来不复杂。但问题很明显：请求频率低的话实时性差，频率高的话浪费服务器资源。而且端口扫描这种场景，扫描到一半卡住了，前端可能还以为在继续扫描。

**第二种是长轮询（Long Polling）**。前端发起请求，服务器如果没有新数据就hold住连接，等到有数据了再返回。这比普通轮询实时性好一些，但实现起来麻烦，而且服务器hold住大量连接的时候资源消耗也不小。

**第三种是 WebSocket**。双向通信，可以服务器主动推送数据。功能最强大，但需要前端引入 socket.io 或者 similar 的库，服务器端也要维护持久连接状态。杀鸡用牛刀。

**第四种是 SSE（Server-Sent Events）**。服务器单向推送数据到客户端，前端用原生的 EventSource API 接收。这正好符合我们的需求：服务器推数据，客户端只管接收和展示。实现简单，代码量少，浏览器原生支持，自动重连。

最终选择了 SSE。选择的主要理由：

1. **单向就够用了**：扫描进度只需要服务器推送到前端，不需要前端主动发数据给服务器（停止扫描用普通 HTTP 请求就够了）
2. **实现简单**：几行代码就能搭起来，不需要引入额外的库
3. **原生支持**：所有现代浏览器都支持 EventSource API，不需要 polyfill
4. **自动重连**：网络断开的时候浏览器会自动重连，不需要自己处理

#### 2.2.2 SSE 的实现细节

Go 标准库没有直接支持 SSE，但 HTTP 响应写文本流的方式天然支持 SSE。SSE 的格式很简单：

```
event: message
data: {"type": "progress", "percent": 50}

event: message
data: {"type": "find", "port": 80}
```

每条消息以 `event:` 开头表示事件类型（可以省略，默认为 message），`data:` 后面跟实际数据，数据可以是 JSON 格式。一条消息以两个换行符结束。

服务端代码：

```go
func sseHandler(w http.ResponseWriter, r *http.Request) {
    // 设置 SSE 响应头
    w.Header().Set("Content-Type", "text/event-stream")
    w.Header().Set("Cache-Control", "no-cache")
    w.Header().Set("Connection", "keep-alive")
    w.Header().Set("Access-Control-Allow-Origin", "*")  // 允许跨域

    // 获取任务 ID
    taskID := r.URL.Query().Get("task")
    task, ok := getTask(taskID)
    if !ok {
        http.Error(w, "Task not found", http.StatusNotFound)
        return
    }

    // 持续监听任务的 channel，有消息就推送给客户端
    flusher, ok := w.(http.Flusher)
    if !ok {
        http.Error(w, "SSE not supported", http.StatusInternalServerError)
        return
    }

    for {
        select {
        case <-r.Context().Done():  // 客户端断开连接
            return
        case msg := <-task.ch:  // 监听任务 channel
            // 写入 SSE 格式的数据
            fmt.Fprintf(w, "event: message\n")
            fmt.Fprintf(w, "data: %s\n\n", mustMarshal(msg))
            flusher.Flush()  // 立即推送给客户端
        case <-task.stop:  // 任务停止
            fmt.Fprintf(w, "event: close\n")
            fmt.Fprintf(w, "data: task stopped\n\n")
            flusher.Flush()
            return
        }
    }
}
```

关键点是 `http.Flusher` 接口。HTTP 响应默认是缓冲的，数据要等到缓冲区满了或者响应结束才会发送到客户端。调用 `Flush()` 可以立即把缓冲区的数据推送给客户端，实现真正的实时推送。

前端代码更简单：

```javascript
// 创建 EventSource 连接
const eventSource = new EventSource(`/sse?task=${taskId}`);

// 监听消息事件
eventSource.onmessage = function(event) {
    const data = JSON.parse(event.data);
    
    switch(data.type) {
        case 'progress':
            updateProgressBar(data.percent);
            break;
        case 'find':
            addResult(data);
            break;
        case 'end':
            eventSource.close();
            break;
    }
};

// 监听错误（网络断开等）
eventSource.onerror = function() {
    console.log('SSE 连接断开，正在重连...');
};

// 主动关闭连接
function stopScan() {
    eventSource.close();
    fetch(`/task/stop?task=${taskId}`, { method: 'POST' });
}
```

#### 2.2.3 踩过的坑

**坑一：IE 浏览器不支持 EventSource**

这个项目本来就不打算支持 IE，所以问题不大。但如果有兼容需求，需要用一个 polyfill 库，比如 event-source-polyfill。

**坑二：Nginx 反向代理默认不转发 SSE**

如果项目部署在 Nginx 后面，需要在 Nginx 配置里添加：

```nginx
location /sse {
    proxy_pass http://backend;
    proxy_http_version 1.1;
    proxy_set_header Connection '';
    proxy_set_header X-Real-IP $remote_addr;
}
```

关键点是 `proxy_set_header Connection ''`（注意后面的空字符串），如果不设置这个头，Nginx 会把连接当成短连接处理，SSE 的长连接会被断开。

**坑三：服务器重启或者进程重启后，客户端的重连行为**

EventSource 会自动重连，但如果服务器端的任务状态没有持久化，重连后的 SSE 连接会拿到新的空任务，跟之前的状态对不上。

我的解决方法是：服务器不维护任务状态，所有状态都在前端维护。SSE 连接只负责推送实时事件，前端根据事件更新本地状态。这样即使重连，也能从最近的已知状态继续（或者干脆从头开始扫描）。

### 2.3 POC 规则格式兼容

#### 2.3.1 三种格式的调研

做 POC 扫描模块之前，我花了不少时间调研现有的 POC 规则格式。

**Legacy 格式**是最简单的，就是定义一个 HTTP 请求的各种参数（URL、方法、请求头、请求体），加上匹配条件（状态码、响应体关键字）。格式大概是这样的：

```yaml
name: ThinkPHP 5.0.23 RCE
method: POST
path: /index.php?s=index/think\app/invokefunction
body: vars[0]=system&vars[1][]=whoami
headers:
  Content-Type: application/x-www-form-urlencoded
validate:
  status: [200]
  body_contains: root
```

优点是简单易懂，缺点是表达能力有限，比如想写"状态码是 200 且响应体包含 'success' 或者包含 'ok'"这种复杂条件就很难表达。

**X-Ray 格式**是长亭科技开源的扫描器使用的格式，最大的特点是有表达式引擎，可以写判断逻辑：

```yaml
id: thinkphp-5023-rce
info:
  name: ThinkPHP 5.0.23 RCE
  severity: high
rules:
  method: POST
  path: /index.php
  body: s=index/think\app/invokefunction
  expression: response.status == 200 && response.body.bcontains(b'root')
```

`expression` 字段里可以写任意的布尔表达式，`response.status` 是响应状态码，`response.body` 是响应体（字节数组），`bcontains(b'root')` 是字节数组合.contains 方法。

这种格式很灵活，可以表达复杂的逻辑，但学习成本稍高，需要了解表达式引擎支持哪些方法和操作符。

**Nuclei 格式**是 ProjectDiscovery 开发的 Nuclei 扫描器使用的格式，特点是规则社区非常活跃，网上有几千条现成的规则。格式相对复杂：

```yaml
id: thinkphp-5023-rce
info:
  name: ThinkPHP 5.0.23 RCE
  author: pdteam
  severity: high
requests:
  - method: POST
    path:
      - "{{BaseURL}}/index.php"
    body: "s=index/think\\app/invokefunction&function=call_user_func_array&vars[0]=system&vars[1][]=whoami"
    matchers:
      - type: word
        words:
          - "root"
          - "/var/www/html"
        condition: or
        part: body
```

Nuclei 格式的 matchers 部分功能很强大，支持 word（关键字匹配）、regex（正则匹配）、status（状态码匹配）、dsl（DSL 表达式匹配）等多种匹配类型，还可以指定匹配的位置（body、header、cookie 等）。

#### 2.3.2 统一解析层的设计

要让这三种格式都能在系统里运行，最直接的做法是写三个解析器，分别解析三种格式。但这会导致代码重复，而且后续维护困难。

我的做法是设计一个统一的中间表示（IR，Intermediate Representation），不管什么格式都先转换成这个中间表示，再统一执行。

中间表示大概是这样的：

```go
type POCRule struct {
    ID      string
    Name    string
    Severity string  // high, medium, low
    Steps   []POCStep
}

type POCStep struct {
    Method     string
    Path       string
    Body       string
    Headers    map[string]string
    Extractors []Extractor  // 变量提取器
    Matchers   []Matcher    // 匹配器
}

type Extractor struct {
    Type     string  // regex, kval, json
```

#### 2.3.3 表达式引擎的实现

X-Ray 格式的 expression 字段是最大的难点，因为需要解析和执行任意的布尔表达式。

一开始想过自己写表达式解析器，但太复杂了。后来找到了一个轻量级的表达式引擎库：github.com/Knetic/govaluate。

这个库可以解析类似这样的表达式字符串：

```
response.status == 200 && response.body.bcontains(b'root')
```

使用方式：

```go
parameters := make(map[string]interface{})
parameters["response.status"] = resp.StatusCode
parameters["response.body"] = resp.Body  // []byte 类型

expression, err := govaluate.NewEvaluableExpression("response.status == 200 && response.body.bcontains(b'root')")
result, err := expression.Evaluate(parameters)
```

但这个库有个限制：自定义函数必须提前注册，而且参数类型要匹配。我需要注册一些自定义函数，比如 `bcontains`、`contains`、`startsWith` 等：

```go
functions := map[string]govaluate.ExpressionFunction{
    "bcontains": func(args ...interface{}) (interface{}, error) {
        // 第一个参数是字节数组，第二个是要查找的内容
        if len(args) < 2 {
            return false, nil
        }
        haystack, ok := args[0].([]byte)
        if !ok {
            return false, errors.New("first argument must be []byte")
        }
        needle, ok := args[1].([]byte)
        if !ok {
            return false, errors.New("second argument must be []byte")
        }
        return bytes.Contains(haystack, needle), nil
    },
    "contains": func(args ...interface{}) (interface{}, error) {
        // 类似 bcontains，但处理字符串
    },
}

expression, _ := govaluate.NewEvaluableExpressionWithFunctions(expressionStr, functions)
```

这样就支持了 X-Ray 格式的表达式匹配。

#### 2.3.4 踩过的坑

**坑一：YAML 解析的大小写问题**

YAML 里的布尔值可以是 `true`、`True`、`TRUE`，解析的时候要统一处理。另外，YAML 里 `yes`、`no` 也可能被解析成布尔值，要小心。

**坑二：Nuclei 的变量替换语法跟 X-Ray 不一样**

Nuclei 用 `{{变量名}}`，X-Ray 用 `{{变量名}}` 但变量来源不同。解析的时候要分别处理。

**坑三：正则表达式的性能问题**

如果规则里用了复杂的正则，匹配可能会很慢。后来加了超时机制，单次匹配超过 5 秒就强制终止。

### 2.4 AI 生成代码的验证机制

#### 2.4.1 为什么需要验证

想让 AI 自动生成漏洞利用代码（EXP），但 AI 第一次生成的代码经常有问题：

1. **语法错误**：缩进不对、括号没配对、引号用了中文等
2. **逻辑错误**：变量名写错、API 调用方式不对
3. **环境问题**：依赖的库没装、Python 版本不兼容
4. **输出格式不对**：返回的数据格式跟我期望的不一样

直接用 AI 生成的代码跑不通，浪费时间和精力。后来想明白了：AI 是生成模型，不是执行环境，它只能保证"看起来对"，不能保证"跑起来对"。所以必须加验证环节。

#### 2.4.2 验证流程的设计

验证流程大概是这样的：

```
1. AI 生成 Python 代码
2. 保存到临时文件
3. 用 Python 执行这个文件
4. 捕获执行过程中的错误和输出
5. 检查输出是否符合预期
   - 如果符合：验证通过，返回生成的代码
   - 如果不符合：分析失败原因
6. 根据失败原因让 AI 修正，或者用规则自动修复
7. 最多重试 5 次
```

关键代码：

```go
func verifyAndFix(pythonCode string, targetURL string) (*VerifyResult, error) {
    for attempt := 1; attempt <= maxRetries; attempt++ {
        // 1. 保存代码到临时文件
        tmpFile := fmt.Sprintf("/tmp/neonscan_exp/exp_attempt_%d.py", attempt)
        err := os.WriteFile(tmpFile, []byte(pythonCode), 0644)
        if err != nil {
            return nil, fmt.Errorf("保存代码失败: %w", err)
        }
        
        // 2. 执行代码，设置超时
        ctx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
        defer cancel()
        
        cmd := exec.CommandContext(ctx, "python3", tmpFile, "--target", targetURL, "--test")
        output, err := cmd.CombinedOutput()
        
        // 3. 分析输出
        result := parseOutput(output, err)
        
        if result.Success {
            // 验证通过
            return &VerifyResult{
                Success:    true,
                PythonCode: pythonCode,
                Attempts:   attempt,
            }, nil
        }
        
        // 4. 验证失败，分析原因
        reason := analyzeFailure(result)
        
        // 5. 根据原因决定下一步
        if reason.CanFixWithRules() {
            // 用规则自动修复
            pythonCode = reason.FixWithRules(pythonCode)
        } else {
            // 让 AI 修正
            pythonCode = requestCorrection(pythonCode, reason)
        }
    }
    
    return nil, fmt.Errorf("超过最大重试次数")
}
```

#### 2.4.3 失败原因分析和修复

常见失败原因大概有以下几类：

**语法错误**：
- 缩进问题（Python 对缩进敏感）
- 括号、引号不配对
- 关键字拼写错误
- 缺少必要的 import

自动修复方法：写一个简单的修复函数，比如自动修正常见缩进问题。

```go
func fixSyntaxError(code string) string {
    // 修复常见缩进问题
    lines := strings.Split(code, "\n")
    var fixed []string
    for _, line := range lines {
        // 如果行尾有不必要的空格，去掉
        line = strings.TrimRight(line, " \t")
        // 如果是注释行，确保前面有正确的缩进
        if strings.HasPrefix(line, "#") && len(fixed) > 0 {
            // 继承上一行的缩进
            indent := getIndent(fixed[len(fixed)-1])
            line = indent + strings.TrimLeft(line, " \t")
        }
        fixed = append(fixed, line)
    }
    return strings.Join(fixed, "\n")
}
```

**输出提取失败**：
- 响应解析方式不对
- 正则表达式匹配不到内容
- 输出格式不符合预期

自动修复方法：修改正则表达式，添加更多的输出标记。

```go
func improveOutputExtraction(code string) string {
    // 确保代码有输出标记
    if !strings.Contains(code, "NEONSCAN_BEGIN") {
        // 在关键输出位置插入标记
        code = strings.Replace(code,
            "print(result)",
            "print('NEONSCAN_BEGIN'); print(result); print('NEONSCAN_END')",
            1)
    }
    return code
}
```

**依赖问题**：
- requests 库没安装
- 其他第三方库缺失

这个没法自动修复，只能提示用户手动安装。但可以在生成代码的时候加上必要的 import 注释：

```python
# 需要先安装依赖：
# pip install requests
import requests
```

当规则修复不了的时候，需要让 AI 帮忙修正。关键是怎么描述问题让 AI 理解。

#### 2.4.5 经验总结

**AI 生成代码不能百分百依赖**：AI 能生成"看起来对"的代码，但不一定能"跑起来对"。必须有验证环节。

**规则修复 + AI 修正的组合拳效果好**：简单的问题用规则自动修复，复杂的问题再让 AI 处理。这样既能保证效率，又能处理边界情况。

**输出格式标准化很重要**：在生成代码的时候就规定好输出格式，比如必须包含特定标记、返回特定结构的 JSON 等。这样验证的时候只需要检查标记就行，不用解析复杂的输出。

### 2.5 MCP 协议集成外部工具

#### 2.5.1 什么是 MCP 协议

MCP（Model Context Protocol）是 Anthropic 提出的一个协议，目标是让 AI 模型能够调用外部工具。简单来说，就是定义一套标准化的"工具发现-工具调用-结果返回"流程。

在这个项目里，我想集成 IDA Pro 做二进制分析、集成 JADX 做 APK 反编译。但这有两个问题：

1. IDA 和 JADX 都是桌面程序，不能直接调用
2. 每个工具的接口都不一样，没法统一处理

MCP 协议解决了这个问题。它定义了一套标准化的接口：

```
客户端（我们的系统）
    │
    │  HTTP/JSON-RPC
    ▼
MCP 服务器（IDA MCP Server / JADX MCP Server）
    │
    │  各自协议
    ▼
IDA Pro / JADX
```

#### 2.5.2 MCP 连接的实现

Go 语言的 MCP 客户端实现大概是这样的：

```go
type MCPClient struct {
    serverURL string
    httpClient *http.Client
}

type MCPRequest struct {
    JSONRPC string          `json:"jsonrpc"`
    Method  string          `json:"method"`
    Params  json.RawMessage `json:"params,omitempty"`
    ID      int             `json:"id"`
}

type MCPResponse struct {
    JSONRPC string          `json:"jsonrpc"`
    Result  json.RawMessage `json:"result,omitempty"`
    Error   *MCPError       `json:"error,omitempty"`
    ID      int             `json:"id"`
}

// 发送请求
func (c *MCPClient) Call(method string, params map[string]interface{}) (json.RawMessage, error) {
    req := MCPRequest{
        JSONRPC: "2.0",
        Method:  method,
        Params:  mustMarshal(params),
        ID:      1,
    }
    
    body, err := json.Marshal(req)
    if err != nil {
        return nil, err
    }
    
    resp, err := c.httpClient.Post(c.serverURL, "application/json", bytes.NewReader(body))
    if err != nil {
        return nil, err
    }
    defer resp.Body.Close()
    
    var mcpResp MCPResponse
    if err := json.NewDecoder(resp.Body).Decode(&mcpResp); err != nil {
        return nil, err
    }
    
    if mcpResp.Error != nil {
        return nil, fmt.Errorf("MCP error: %s", mcpResp.Error.Message)
    }
    
    return mcpResp.Result, nil
}
```

调用 MCP 工具的方式很简单：

```javascript
class ScanManager {
    constructor() {
        this.currentTaskId = null;
        this.eventSource = null;
    }
    
    startScan(type, params) {
        // 先发送请求启动扫描任务
        fetch(`/scan/${type}`, {
            method: 'POST',
            headers: {'Content-Type': 'application/json'},
            body: JSON.stringify(params)
        })
        .then(resp => resp.json())
        .then(data => {
            this.currentTaskId = data.taskId;
            this.connectSSE(data.taskId);
        });
    }
    
    connectSSE(taskId) {
        // 建立 SSE 连接
        this.eventSource = new EventSource(`/sse?task=${taskId}`);
        
        this.eventSource.onmessage = (event) => {
            const msg = JSON.parse(event.data);
            this.handleMessage(msg);
        };
        
        this.eventSource.onerror = () => {
            console.log('SSE 连接断开');
            // 自动重连
            setTimeout(() => {
                if (this.currentTaskId) {
                    this.connectSSE(this.currentTaskId);
                }
            }, 3000);
        };
    }
    
    handleMessage(msg) {
        switch(msg.type) {
            case 'start':
                this.onScanStart(msg);
                break;
            case 'progress':
                this.onProgress(msg);
                break;
            case 'find':
                this.onFind(msg);
                break;
            case 'end':
                this.onScanEnd(msg);
                this.eventSource.close();
                break;
        }
    }
    
    onProgress(msg) {
        // 更新进度条
        const percent = msg.percent;
        document.getElementById('progress-bar').style.width = percent + '%';
        document.getElementById('progress-text').textContent = `${msg.progress} (${percent}%)`;
    }
    
    onFind(msg) {
        // 添加发现结果
        const resultsDiv = document.getElementById('results');
        const item = document.createElement('div');
        item.className = 'result-item';
        item.textContent = JSON.stringify(msg.data);
        resultsDiv.appendChild(item);
    }
    
    stopScan() {
        if (this.currentTaskId) {
            fetch(`/task/stop?task=${this.currentTaskId}`, {method: 'POST'});
            this.eventSource.close();
        }
    }
}
```

### 2.7 文件上传安全

#### 2.7.1 安全风险识别

文件上传功能有几个潜在的安全风险：

1. **恶意文件**：用户可能上传可执行脚本、webshell 等
2. **大文件**：用户可能上传超大文件耗尽服务器存储
3. **路径遍历**：文件名可能包含 `../` 之类的路径遍历字符
4. **文件覆盖**：同名文件可能被覆盖

#### 2.7.2 防护措施

**措施一：文件大小限制**

```go
const maxFileSize = 512 * 1024 * 1024  // 512MB

func uploadHandler(w http.ResponseWriter, r *http.Request) {
    // 解析 multipart 表单
    err := r.ParseMultipartForm(maxFileSize)
    if err != nil {
        http.Error(w, "文件太大", http.StatusBadRequest)
        return
    }
    
    file, header, err := r.FormFile("file")
    if err != nil {
        http.Error(w, "读取文件失败", http.StatusBadRequest)
        return
    }
    defer file.Close()
    
    // 文件大小已经在 ParseMultipartForm 里检查过了
    // 但还是要再检查一次更保险
    if header.Size > maxFileSize {
        http.Error(w, "文件太大", http.StatusBadRequest)
        return
    }
}
```

**措施二：文件名安全处理**

```go
func safeFilename(filename string) string {
    // 过滤掉路径遍历字符
    filename = filepath.Base(filename)  // 只保留文件名部分
    filename = strings.ReplaceAll(filename, "..", "")
    
    // 替换特殊字符
    filename = regexp.MustCompile(`[^\w.-]`).ReplaceAllString(filename, "_")
    
    // 加时间戳防止重复
    ext := filepath.Ext(filename)
    name := filename[:len(filename)-len(ext)]
    return fmt.Sprintf("%s_%d%s", name, time.Now().Unix(), ext)
}
```

**措施三：隔离存储**

```go
func saveUploadedFile(file *multipart.FileHeader) (string, error) {
    // 在 uploads 目录下按时间戳创建子目录
    uploadDir := fmt.Sprintf("uploads/u-%d/", time.Now().UnixNano()/1e6)
    if err := os.MkdirAll(uploadDir, 0755); err != nil {
        return "", err
    }
    
    // 保存文件
    dstPath := filepath.Join(uploadDir, safeFilename(file.Filename))
    srcFile, err := file.Open()
    if err != nil {
        return "", err
    }
    defer srcFile.Close()
    
    dstFile, err := os.Create(dstPath)
    if err != nil {
        return "", err
    }
    defer dstFile.Close()
    
    if _, err := io.Copy(dstFile, srcFile); err != nil {
        return "", err
    }
    
    return dstPath, nil
}
```

**措施四：定期清理**

写了一个清理函数，在后台定期执行：

```go
func startJanitor() {
    ticker := time.NewTicker(1 * time.Hour)
    go func() {
        for range ticker.C {
            cleanupOldUploads()
        }
    }()
}

func cleanupOldUploads() {
    maxAge := 7 * 24 * time.Hour  // 7 天
    cutoff := time.Now().Add(-maxAge)
    
    filepath.Walk("uploads", func(path string, info os.FileInfo, err error) error {
        if err != nil {
            return err
        }
        if info.IsDir() && strings.HasPrefix(info.Name(), "u-") {
            // 检查目录创建时间
            if info.ModTime().Before(cutoff) {
                os.RemoveAll(path)  // 删除整个目录
            }
        }
        return nil
    })
}
```

## 第三部分：技术选型的深层思考

### 3.1 为什么选择 Go 而不是 Python

在确定用 Go 之前，我犹豫了很久。毕竟 Python 写代码快，库也多，如果用 Python 的话很多功能可以直接调用现成的库，不用自己造轮子。

但最终让我下定决心的有几个原因：

**第一个是性能**。安全扫描本质上是 I/O 密集型任务，要同时探测大量端口和 URL。Python 有 GIL，虽然有多线程库，但实际运行的时候还是单核。Go 的 goroutine 是真正的多核并发，可以充分利用多核 CPU 的优势。

我做过一个简单的对比测试：用 Python 和 Go 分别写一个端口扫描器，扫描 1000 个端口。Python 版本耗时约 45 秒，Go 版本耗时约 12 秒，差了将近 4 倍。

**第二个是部署**。Python 项目部署的时候要考虑环境问题：目标服务器是 Linux 还是 Windows、Python 版本是多少、有没有装 pip、国内服务器访问 PyPI 慢不慢……Go 编译成单个二进制文件，直接拷贝过去就能跑，不存在这些问题。

**第三个是内存**。Python 跑久了内存会越来越高，因为有内存碎片和循环引用的问题。Go 有垃圾回收，但比 Python 高效很多，而且 goroutine 的栈空间是动态管理的，不会像 Python 那样每个线程都占用固定大小的栈内存。

**代价是学习成本**。Go 语言花了我大概两个月才基本熟练，Python 的话可能两周就够了。但考虑到项目要用好几年，学习成本是一次性的，值得投入。

### 3.2 为什么不用前端框架

现代前端开发基本上离不开 Vue、React 这些框架，我也纠结过要不要用。

**最终选择原生 HTML/CSS/JS 的原因**：

**第一是项目规模**。这个项目虽然功能多，但前端代码量其实不大，所有页面加起来也就几千行。引入 Vue 的话，需要配置构建工具（Webpack 或者 Vite）、理解组件化思想、解决状态管理问题……这些工作量不小，但对于这个规模的项目来说收益不大。

**第二是依赖管理**。用 Vue 的话，package.json 里有几十个依赖，每个依赖又可能有自己的依赖，形成复杂的依赖树。依赖多了之后，版本冲突、更新维护都是问题。原生 JS 没有这个问题，所有代码都在眼前。

**第三是调试**。原生 JS 出问题了可以直接在浏览器里打断点，Vue 的话有自己的一套运行机制，问题可能出在模板编译、响应式系统、还是业务逻辑，有时候不太好定位。

**不用的理由**：

对于安全工具来说，前端不是重点，不需要多么炫酷的交互效果。能清晰展示扫描结果、能实时显示进度、能有基本的表单交互就够了。过度设计反而增加复杂度。

### 3.3 SSE 选型的考量

选择 SSE 而不是 WebSocket，主要是从实际需求出发的：

**需求分析**：扫描进度推送只需要服务器单向发数据给浏览器，浏览器不需要主动发数据给服务器（停止扫描用普通的 HTTP POST 就够了）。这是一个典型的"一对多"的发布订阅场景。

**方案对比**：

| 方案 | 实现复杂度 | 能力 | 前端依赖 | 服务器资源 |
|------|-----------|------|----------|------------|
| 轮询 | 低 | 弱，实时性差 | 无 | 高（频繁请求） |
| 长轮询 | 中 | 中 | 无 | 中 |
| WebSocket | 高 | 强 | 需要库 | 低 |
| SSE | 低 | 中（单向） | 无 | 低 |

WebSocket 功能更强大，但需要额外引入库（socket.io 或者原生 WebSocket 虽然浏览器支持但 IE 不支持），服务器端也要维护连接状态。对于只需要单向推送的场景，SSE 更合适。

**SSE 的限制**：

SSE 是基于 HTTP/1.1 的，一个 TCP 连接上只能有一个 SSE 会话。如果需要双向通信，或者在同一个页面上建立多个 SSE 连接，WebSocket 是更好的选择。

### 3.4 POC 格式兼容的权衡

一开始我只想支持一种格式，但后来发现这样做问题很大。

**只用 Legacy 格式的问题**：简单是简单了，但很多优秀的 POC 规则是 X-Ray 或者 Nuclei 格式的，直接拿来用不行。

**只用 Nuclei 格式的问题**：Nuclei 格式最完善，但解析逻辑复杂，而且 Nuclei 的 DSL 表达式跟 X-Ray 的表达式语法不一样。

**最终方案**：支持三种格式，但在实现上做了权衡：

- Legacy 格式：完整支持，解析和执行都简单
- X-Ray 格式：支持 expression 表达式引擎，但不支持高级特性（比如 requestor 之类）
- Nuclei 格式：支持常用的 matchers，但不支持复杂的 workflows 和子模板

这样既能复用大部分现有规则，又不会把解析逻辑搞得太复杂。

## 第四部分：开发过程中的反思

### 4.1 时间管理的问题

这个项目最大的问题可能是时间管理。一开始觉得功能不多，应该很快能做完，结果发现每个功能都比预想的要复杂。

**问题一：低估了"磨细节"的时间**

核心功能写出来可能只需要一周，但要让这个功能稳定可用、错误处理完善、边界情况考虑周全，可能需要两到三周。

比如端口扫描，写一个能跑起来的版本可能两天就够了。但要处理各种异常情况：网络超时怎么处理、目标不可达怎么办、用户输入错误怎么提示……这些细节加起来花的时间比写核心功能还多。

**问题二：调研时间超出预期**

有些功能之前没接触过，要先调研方案、读文档、看示例代码。这部分时间往往被低估。

比如 POC 格式兼容，我一开始觉得解析 YAML 就行了，结果发现每种格式的语法都不一样，表达式引擎也很复杂，前后花了将近一个月才做到基本可用。

**问题三：需求变更**

开发过程中，导师提了一些改进意见，有些是合理的需求，有些是看了别人家的产品后想加的功能。变更多了之后，进度就delay了。

**经验教训**：

1. 核心功能先做"能用"，再考虑"好用"
2. 调研的时候多留一些 buffer 时间
3. 需求变更要控制，可以记录下来放到 v2

### 4.2 代码质量的问题

作为一个学生项目，这个项目的代码质量参差不齐，有些地方写得比较糙。

**问题一：错误处理不完善**

很多地方直接 `panic` 或者 `log.Fatal`，没有优雅地处理错误。比如网络请求失败的时候，应该重试几次而不是直接退出。

**问题二：注释不够**

写代码的时候觉得逻辑很清楚，不用写注释。过几个月回头看，完全不知道自己当时想表达什么。代码应该"自解释"，但适当的注释能省很多理解成本。

**问题三：单元测试覆盖率低**

几乎没有写单元测试，主要靠手动测试。这在项目小的时候没问题，但项目大了之后，改一个地方可能影响另一个地方，没有测试心里没底。

**经验教训**：

1. 代码规范要提前定好，不能想怎么写就怎么写
2. 注释要写，但写"为什么"而不是"是什么"
3. 核心模块要有单元测试，哪怕是最简单的测试

### 4.3 技术债务

由于时间压力，有些代码是"先跑起来再说"的姿态写的，积累了一些技术债务：

**债务一：重复代码**

几个扫描模块里有不少相似的代码，比如 HTTP 请求发送、结果解析等。本来应该抽成公共函数，但当时为了赶进度直接复制粘贴了。

**债务二：硬编码**

有些参数是硬编码的，比如超时时间、并发数等。更好的做法是写成配置文件或者命令行参数。

**债务三：日志不完善**

有些关键操作没有记录日志，出问题的时候很难排查。应该是关键路径上都有适当的日志输出。

**债务四：没有抽象层**

代码基本上是面向过程写的，模块之间耦合比较紧。如果以后要换存储后端或者换 AI 提供商，改动会比较大。

**怎么处理**：

这些问题短期不影响使用，但长期会阻碍项目发展。我的策略是：

1. 新增功能按照规范写，不欠新债
2. 改动旧功能的时候顺便重构，不积累
3. 大的重构等到项目相对稳定之后再做

## 第五部分：对未来改进的思考

### 5.1 功能扩展

**子域名扫描**：目前只支持 IP 和域名的扫描，没有子域名发现功能。这个功能在实战中很有用。

**密码爆破**：对于登录接口，可以做用户名密码的组合爆破，当然要有速率控制防止锁号。

**流量分析**：可以加入被动流量分析的能力，比如抓包识别敏感接口。

**报告增强**：目前只能导出 Markdown 格式的报告，希望能支持 PDF、HTML 等格式，并且可以在线分享。

### 5.2 性能优化

**分布式架构**：目前是单机版，如果有大量目标要扫描，性能会受限。可以考虑用消息队列做任务分发，支持多节点协作。

**结果缓存**：对于相同的扫描目标，可以缓存扫描结果，避免重复扫描。

**智能并发**：目前并发数是固定配置的，以后可以根据目标响应情况动态调整：响应快的可以加大并发，响应慢的就减少并发。

### 5.3 安全加固

**用户认证**：目前没有用户系统，所有人共用一个界面。以后可以加入用户认证和权限管理。

**操作审计**：目前只记录到日志文件，以后可以考虑存到数据库，方便查询和统计。

**数据加密**：敏感数据（API 密钥等）目前只存内存不落盘，但日志里可能有敏感信息，需要脱敏处理。

### 5.4 用户体验

**响应式设计**：目前页面没有做移动端适配，手机上几乎没法用。

**主题切换**：可以加一个浅色/深色主题切换的功能。

**国际化**：界面文字目前都是中文硬编码，以后如果要支持多语言，需要做 i18n 处理。

## 第六部分：答辩经验分享

### 6.1 答辩前的准备

答辩之前我把项目的各个方面都梳理了一遍：

**功能演示**：确保每个功能都能跑起来，准备好几个典型的演示案例。答辩现场很可能出问题，所以要准备 Plan B、Plan C。

**代码结构**：对自己写的代码要熟悉，评委可能问为什么要这样设计、那里为什么不那样做。

**技术细节**：对自己用的技术要有理解，不能只说"我用了这个库"，要能说出"为什么选这个库"、"它解决了什么问题"。

**可能的问题**：提前想好评委可能会问的问题，准备好回答。

### 6.2 常见问题和建议

**Q: 为什么选择这个技术？**

不要只说"因为这个技术很火"，要结合项目需求说。比如选择 Go，可以说"因为安全扫描需要高并发，Go 的 goroutine 比 Python 的多线程性能好很多，实测扫描速度快了 4 倍"。

**Q: 项目有哪些创新点？**

可以从几个角度说：一体化整合（别人是分散的工具，我们是统一的平台）、数据闭环（别人的数据不流转，我们能形成完整的数据流）、智能化（别人是纯扫描，我们有 AI 分析能力）。

**Q: 遇到的最大难点是什么？**

找一个真实有挑战的难点说，然后重点讲清楚是怎么解决的。比如"POC 格式兼容"，难点是"三种格式语法不一样"，解决方案是"设计统一的中间表示，用适配器模式转换"。

**Q: 如果继续做下去会怎么优化？**

说实话项目的不足，不要逞强说"已经很完美了"。可以说性能优化、功能扩展、代码重构这些方向。

### 6.3 答辩现场的感受

答辩的时候确实会紧张，但最重要的还是**对自己的项目有信心**。你可能不是所有问题都能答上来，但只要你对自己做的东西足够了解，评委是能感受到的。

评委一般不会刻意为难学生，他们只是想确认这个项目是不是你自己做的、你有没有真正学到东西。所以如实回答就好，不懂的问题可以说"这个我没深入研究过"。

还有一点，演示的时候如果出了问题不要慌。我在演示 AI 分析功能的时候，AI 接口响应超时了，大屏幕上卡了好几秒。我就说"网络有点问题，我们跳过这个环节，直接看静态报告"，然后翻到之前保存的截图。灵活应变比硬撑着更重要。
